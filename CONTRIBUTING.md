# Contributing to BarsikCMD 🐾

Спасибо, что хотите помочь развитию **BarsikCMD**.  
Любой вклад — от исправления опечатки до добавления новой команды — важен.

Этот документ объясняет, как участвовать в проекте.

---

## Как начать

1. Сделайте **fork** репозитория.
2. Создайте новую ветку:
   ```bash
   git checkout -b my-change-name
   ```
3. Внесите изменения.
4. Убедитесь, что всё собирается и запускается.
5. Создайте **pull request** с понятным описанием.

Если вы не уверены, всё ли сделали правильно — это нормально. В PR всегда можно обсудить детали.

---

## Pull Requests

Чтобы PR было проще принять:

- Делайте изменения логичными и не слишком огромными.
- В описании PR укажите:
  - что именно изменено,
  - зачем это сделано,
  - есть ли побочные эффекты.
- Если PR исправляет баг — опишите, как его воспроизвести.
- Если PR добавляет новую команду — обновите README или Wiki (если это уместно).

PR, которые сильно усложняют BarsikCMD без реальной пользы, могут быть отклонены.

---

## Коммиты

Специального формата нет — важнее **понятность**.

Рекомендации:

- Заголовок коммита — коротко и по делу:
  - `Исправлен баг с update`
  - `Добавлена команда echo`
  - `Обновлён README`
- В полном описании (body) можно расписать детали:
  - что именно поменяли,
  - почему так,
  - если есть нюансы — указать их.

Главная цель — чтобы по истории коммитов было понятно, что происходило с проектом.

---

## Код‑стиль

BarsikCMD написан на **C и C++**.

Общие принципы:

- следуйте уже существующему стилю в проекте;
- избегайте лишних зависимостей;
- пишите максимально простой и читаемый код;
- не добавляйте «магии», если можно сделать явно.

BarsikCMD ценит минимализм и предсказуемость.

---

## Тестирование

Перед отправкой PR:

- убедитесь, что проект компилируется;
- проверьте, что базовые команды (`help`, `ver`, `echo`, `update`, `clear`, `exit`) работают;
- если вы исправляете конкретный баг — проверьте, что он действительно исчез.

---

## Issues

Issues можно использовать для:

- сообщений об ошибках,
- предложений новых фич,
- вопросов,
- идей на будущее.

Перед созданием issue:

- посмотрите, нет ли уже похожей темы;
- опишите проблему или идею максимально понятно.

---

## Что не приветствуется

- добавление телеметрии или скрытого сбора данных;
- изменения, которые оставляют мусор в системе;
- чрезмерное усложнение кода без реальной необходимости;
- PR без описания.

---
